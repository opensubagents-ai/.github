# opensubagents-ai — 7-Workstream Initiative 2026-Q3 Master PRD

## Introduction

opensubagents-ai is moving from a published architecture report and a 22-commit / 17-repo plan into execution. This master PRD covers the 7 workstreams of the 2026-Q3 initiative: the cross-cutting PM substrate, the subprocessor whitelist enforcement, and the upstream-tracking discipline. It is one of two parallel PRDs in this directory; the other is `prd.txt` (the dq-subagent quality-gate PRD from 2026-05-23 loops 1-5).

## Product Overview

opensubagents-ai is an open-by-default organization of polyrepo Claude-Code-native software. Each repo has a single tightly-scoped purpose. The 20 repos are organized in three concentric rings:

- **Inner ring (5 repos):** `sdk`, `agent-sdk`, `cli`, `skills`, `playbooks` — primitives consumed everywhere
- **Middle ring (8 repos):** `mcp-monitoring`, `mcp-business-ops`, `mcp-knowledge`, `channels-webhook-router`, `channels-monitoring`, `agents-orchestrator`, `agents-rubrics`, `infra` — capability surfaces
- **Outer ring (7 repos):** `audit-log`, `trace-collector`, `claude-chrome-sidecar`, `claude-chrome-evaluator`, `standup-agent`, `standup-evaluator`, plus the two new private marketplaces `subagent-plugins` and `coworker-plugins` — observability + runtime + plugin catalogs

This PRD does NOT create new repos. Every workstream fits one of the 20 existing repos.

## Goals

1. Stand up Task Master AI (standard mode, Claude Code models) as the PRD-to-tasks layer.
2. Mirror every Task Master task into Jira project `OP` via Rovo MCP; mirror every Jira story into GitHub Project v2.
3. Adopt `anthropics/claude-agent-sdk-demos/research-agent` and `email-agent` verbatim with biweekly upstream sync.
4. Stand up the design pipeline (cmark-gfm + unified/rehype + @vscode/html-languageservice) mirroring `anthropics/knowledge-work-plugins/design`.
5. Stand up the data plane (Docker Business + AlloyDB Omni 16.8.0 + DragonflyDB latest + Docker MCP profiles).
6. Wire Cloudflare GraphQL Analytics through `mcp-monitoring` using `graphql.mcp.cloudflare.com`.
7. Deploy `opensubagents.com` (marketing) + `outcomesdk.com` (developer surface) on Cloudflare with security headers and email auth.
8. Capture $100K+ in stacked startup credits via the email-agent and the operator's promo-credits routine.

## Non-Goals

- Do NOT create new repos. Every workstream fits one of the 20 existing repos.
- Do NOT roll custom MCP protocol code — use `modelcontextprotocol/` SDKs.
- Do NOT bypass the subprocessor whitelist. Non-whitelist vendors require an ADR.
- Do NOT translate Python upstream demos to TypeScript — vendor as-is.
- Do NOT block on credit applications. The engineering work proceeds whether or not credits land.
- Do NOT interfere with the dq-subagent work (separate PRD at `prd.txt`); the two initiatives are independent and can ship in parallel.
- Do NOT create a new Jira project; the `OP` project (name: `opensubagents`) already exists and is the canonical home for all 7 workstreams' epics.

## User Stories and Acceptance Criteria

### ST-101: Operator parses workstream PRDs

As the operator, I want to run `task-master parse-prd` on each workstream PRD so that each workstream has a generated task list.

**Acceptance:** all 7 workstream PRDs are parsed; each emits a populated `.taskmaster/tasks/tasks.json` per tag; `task-master list --tag=<workstream>` shows tasks for each.

### ST-102: Operator (or desktop-agent) mirrors tasks to Jira `OP`

As the operator or desktop-agent, I want every Task Master task to have a corresponding Jira story in project `OP` so that the team can see the work in Jira.

**Acceptance:** `infra/scripts/tm-jira-sync.ts` runs on `.taskmaster/tasks/**` changes; every task has a populated `jira_key` field (format `OP-NNNN`) within 60s of creation; drift between Task Master `status` and Jira `status` is ≤ 5 minutes.

### ST-103: Operator submits Anthropic Startups credit application

As the operator, I want a drafted Anthropic Startups application via the email-agent so that I can review and send it.

**Acceptance:** `marketing-agent` produces a draft in Gmail Drafts with subject, recipient, and body; the draft is labeled `OSA/anthropic-startups`; SQLite `application_status` row exists with status `drafted`; operator reviews and either approves (status → `sent`) or rejects (status → `rejected`).

### ST-104: A WS5 PR introducing DragonflyDB cannot merge without dragonfly_compat.test.ts passing

As a reviewer, I want CI to fail the merge if Redis compatibility tests fail so that we never accidentally regress on EVAL or SCAN semantics.

**Acceptance:** `audit-log/tests/dragonfly_compat.test.ts` is a required check in branch protection on `audit-log` and `trace-collector`.

### ST-105: A WS6 commit lands cloudflare-graphql queries cleanly

As an engineering IC, I want the CF GraphQL queries to be valid against the schema so that I can run them.

**Acceptance:** baseline queries (one per documented dataset) live in `mcp-monitoring/src/integrations/cloudflare-graphql/queries/`; CI runs `graphql_schema_search` against each and confirms the response types match.

### ST-106: Non-whitelist vendor adoption is blocked without ADR

As the operator, I want PRs that introduce non-whitelist vendors to be blocked until the ADR is accepted so that we never accidentally onboard a vendor without review.

**Acceptance:** CI on every repo checks new files in `infra/docs/decisions/` for vendor mentions; PRs that add `package.json` deps or `docker-compose.yml` services pointing to non-whitelist vendors are flagged.

## Technical Requirements / Stack

- **Language:** TypeScript everywhere except `agent-sdk/examples/research-agent/` (Python; documented exception via per-repo `CLAUDE.md`)
- **Package manager:** `pnpm` (consistency across the org)
- **Runtime:** Node 22 LTS on local + CI; Docker images use Node 22 on DHI base
- **MCP SDKs:** `modelcontextprotocol/typescript-sdk` and `modelcontextprotocol/python-sdk` only
- **CI:** GitHub Actions; workflow files cannot land via desktop App connector (Workflows:write blocked) — route via `/handoff` from the `handoff-to-sidecar` plugin
- **Commit format:** Conventional Commits with outcome-tag suffix per `.github/CONVENTIONS.md`
- **Merge strategy:** Squash-merge; delete branch on merge; direct-to-main only for `claude-chrome-sidecar/{tasks,sessions,memory}/`
- **License:** Apache-2.0 for org-original code; MIT preserved for vendored upstreams; commercial vendors per their license

## Design and User Interface

- **Marketing site (opensubagents.com):** static HTML/CSS baseline via Cloudflare Pages; design system mirrors the `anthropics/knowledge-work-plugins/design` plugin's structure (7 skills); brand baseline pulled from `agentknowledgeworkers.com` via headless browser (Playwright or Cloudflare Browser Rendering MCP) once.
- **Developer surface (outcomesdk.com):** docs site via Cloudflare Pages by default; Mintlify only with ADR 0010.
- **PM dashboards:** GitHub Project v2 board `opensubagents-ai-2026Q3` (primary); Jira project `OP` dashboards (secondary).
- **No custom CMS:** prose lives in markdown in repos; the design renderer compiles markdown to brand-consistent HTML.

## Cross-cutting infra

- modelcontextprotocol/ SDKs for everything MCP-flavored
- Conventional Commits + squash-merge via CODEOWNERS + branch protection
- All upstream-vendored code lives behind a `chore: sync from upstream` workflow on biweekly cron
